# DHU履修ガイダンスポータル 管理・運用設計メモ

> 作成: 2026-03-06

## 現状の構造

```
Google Spreadsheet (シラバス)
    ↓ npm run sync
syllabi.json → syllabi.js → App.jsx（シラバスタブ）
                                   ├→ キャリアマップ（IDで参照）
                                   ├→ チャット（キーワードマッチ）
                                   └→ スケジュールタブ

faqs.js ──── 手書き ──→ FAQタブ / チャット
careers.js ── 手書き ──→ キャリアマップタブ
schedule.json ── sync ──→ スケジュールタブ
```

### 課題

- 同じ情報（履修登録期間、科目名など）が `faqs.js` `syllabi.js` `schedule.json` に散在
- 1箇所直しても他が古いまま残るリスク
- FAQ・キャリアマップの更新にコード編集が必要

---

## 提案1: データのシングルソース化（最重要）

**全データをGoogle Spreadsheetに集約し、`npm run sync` 1発で全部取得する**

| シート | 内容 | 現状 |
|---|---|---|
| シラバス | 58科目の全情報 | ✅ 済み |
| スケジュール | 開講日程 | ✅ 済み |
| FAQ | 質問＋回答 | ❌ 手書き（faqs.js） |
| キャリアマップ | 20パス＋推奨科目ID＋理由 | ❌ 手書き（careers.js） |
| 共通設定 | 年度、履修登録期間、各種締切 | ❌ 未対応 |

### 実現可能性: 高

既存のsyncスクリプトを拡張するだけ。

### FAQ内の変数展開

FAQ回答テンプレートに `{{履修登録開始日}}` のような変数を埋め込み、syncスクリプトで共通設定シートから値を解決する仕組みにすると、日付変更時にFAQを個別修正する必要がなくなる。

## 提案2: 年度切替の仕組み

### 設定ファイル案

```json
{
  "year": "2026",
  "spreadsheetId": "1w76UXSc...",
  "registrationStart": "2026-04-07",
  "registrationEnd": "2026-04-18"
}
```

### 年度替わりの作業フロー

1. 新年度のスプレッドシートを作成（前年度をコピー → 内容更新）
2. `config.json` の `spreadsheetId` と年度を書き換え
3. `npm run sync && npm run build` → デプロイ

### 実現可能性: 高

既存の仕組みの延長。

### キャリアマップの年度またぎ問題

推奨科目はIDで参照しているため、年度ごとにIDが変わると壊れる。対策:

- **案A**: IDを科目名ベースのキー（例: `"contents-design-b"`）に変える
- **案B**: 名前マッチングで自動紐付け（科目名はほぼ変わらない前提）
- **案C**: IDを年度間で固定するルールを決める（現実的にはこれで十分）

## 提案3: 管理用スクリプト群

| スクリプト | 内容 | 優先度 |
|---|---|---|
| `npm run sync` | 全データ一括取得（シラバス＋スケジュール＋FAQ＋キャリアマップ＋共通設定） | 高 |
| `npm run validate` | データ整合性チェック（壊れたID参照、空フィールド、重複ID検出） | 高 |
| `npm run diff` | 前回syncとのデータ差分を表示（追加/削除/変更された科目等） | 中 |
| `npm run export-pdf` | シラバス全科目をPDF化（Puppeteer等、ガイダンス配布用） | 中 |

### validate の具体的チェック項目

- キャリアマップの推奨科目IDがシラバスに存在するか
- FAQの日付が共通設定と一致しているか
- シラバスに必須フィールド（名前、クォーター、担当教員等）の欠落がないか
- スケジュールの日付フォーマットが正しいか
- 重複IDの検出

## 提案4: 管理ダッシュボードは必要か？

**結論: 現時点では不要**

- 利用者（管理者）が1人であればスプレッドシート＋CLIスクリプトのほうが軽量で保守コストが低い
- 複数人が運営するようになったら検討

ただし、もし作るなら:

| 機能 | 実現可能性 |
|---|---|
| データプレビュー（sync後デプロイ前の確認） | 高（`npm run dev` で既に可能） |
| 差分レポート（前回との変更点表示） | 中（sync時にdiff出力） |
| 個別科目の上書き編集（スプレッドシート触らず修正） | 中（overrides.json方式） |
| ワンクリックデプロイ | 低（GitHub Actions連携） |

## 提案5: 理想的な運用フロー

```
スプレッドシートで修正
    ↓
npm run sync     ← 全データ取得
    ↓
npm run validate ← 整合性チェック（エラーがあれば警告）
    ↓
npm run dev      ← ローカルで目視確認
    ↓
npm run build && npm run deploy
```

### これを実現するために必要な開発

| 項目 | 作業量の目安 |
|---|---|
| FAQデータのスプレッドシート化 | スプレッドシート設計 + sync拡張 |
| キャリアマップのスプレッドシート化 | スプレッドシート設計 + sync拡張 |
| 共通設定シートの作成 | シート設計 + sync拡張 + FAQ変数展開 |
| validateスクリプトの新設 | 新規作成 |
| 年度切替手順の文書化 | ドキュメントのみ |

## 優先順位まとめ

| 順 | やること | 効果 |
|---|---|---|
| 1 | 共通設定シートの設計 + FAQ/キャリアマップのスプレッドシート化 | 情報の一元管理 |
| 2 | syncスクリプトの拡張（全データ一括取得） | 更新作業の簡素化 |
| 3 | validateスクリプトの新設 | ミス防止 |
| 4 | 年度切替手順の文書化 | 引き継ぎ可能性 |
| 5 | PDF出力・差分表示等の便利機能 | あれば嬉しい |
